# 4.4 有界宽容原则

[← 返回文档导航](../00-文档导航.md)

---

## 📖 什么是"有界宽容原则"？

这是 JSONMap 和 ValUtil 的核心设计原则，在宽容与严格之间找到平衡：

**有宽容**：对预期内的情况保持宽容
- ✅ 缺失容忍：路径不存在
- ✅ 类型容忍：类型可转换（"1" → 数字）

**有边界**：对预期外的错误严格失败
- ❌ 内容不容忍：内容无法转换（"abc" → 数字）

---

## 🎯 三大核心规则

### 1. 缺失容忍（Lenient on Missing）

**路径不存在时返回 null 或默认值，不抛异常**

```java
JSONMap data = new JSONMap("{\"user\":{\"name\":\"张三\"}}");

// 路径不存在，返回 null
String age = data.getStr("user.age");           // → null ✅
String city = data.getStr("user.profile.city"); // → null ✅

// 使用默认值
String city = data.getStr("user.profile.city", "未知");  // → "未知"
```

### 2. 类型容忍（Lenient on Type）

**类型不同但能转换时，自动转换**

```java
// ✅ 类型不同但能转换：自动转换
ValUtil.toInt("123");          // String → Integer ✅
ValUtil.toInt(123L);           // Long → Integer ✅
ValUtil.toDouble("99.9");      // String → Double ✅
ValUtil.toBoolean("true");     // String → Boolean ✅
ValUtil.toBoolean(1);          // Integer → Boolean ✅
```

### 3. 内容不容忍（Strict on Content）

**内容无法转换时，立即抛异常（快速失败）**

```java
// ❌ 内容无法转换：抛出异常
ValUtil.toInt("abc");          // → NumberFormatException ❌
ValUtil.toInt("xyz");          // → NumberFormatException ❌
ValUtil.toDouble("hello");     // → NumberFormatException ❌

// ✅ null 或空字符串：返回 null 或默认值（这是缺失，不是内容错误）
ValUtil.toInt(null);           // → null
ValUtil.toInt(null, 0);        // → 0
ValUtil.toInt("");             // → null
ValUtil.toInt("", 0);          // → 0
```

---

## 📊 规则总结表

| 情况 | 分类 | 行为 | 示例 |
|------|------|------|------|
| **路径不存在** | 缺失容忍 | 返回 null 或默认值 | `data.getStr("user.age")` → null |
| **值为 null** | 缺失容忍 | 返回 null 或默认值 | `ValUtil.toInt(null, 0)` → 0 |
| **值为空字符串** | 缺失容忍 | 返回 null 或默认值 | `ValUtil.toInt("", 0)` → 0 |
| **类型不同但能转换** | 类型容忍 | 自动转换 | `ValUtil.toInt("123")` → 123 |
| **内容无法转换** | 内容不容忍 | 抛出异常 | `ValUtil.toInt("abc")` → 异常 |

---
## 🤔 为什么要"有界宽容"？

### 为什么要"宽容"？

**1. 缺失容忍：数据结构可能不完整**

```java
// API 返回的数据可能缺少某些字段
JSONMap response = new JSONMap(apiResponse);
String nickname = response.getStr("user.profile.nickname");  // 可能不存在，返回 null ✅
```

**2. 类型容忍：前端传参类型混乱**

```java
// 前端可能传 "123" 也可能传 123
Integer age = ValUtil.toInt(request.getParameter("age"));  // 都能转换 ✅
```

### 为什么要"有界"（内容不容忍）？

**1. 及早发现问题**

```java
// ❌ 静默失败 - 问题被隐藏
Integer age = ValUtil.toInt("abc", 0);  // 返回 0，但 "abc" 根本不是数字
saveUser(age);  // 保存了错误数据，问题延后到数据库或业务逻辑

// ✅ 快速失败 - 问题立即暴露
Integer age = ValUtil.toInt("abc");  // 立即抛出 NumberFormatException
// 开发者马上知道数据有问题，可以立即修复
```

**2. 明确的契约**

```java
// 方法契约：toInt() 将对象转换为 Integer
// - 如果对象可以转换 → 返回 Integer
// - 如果对象不能转换 → 抛出异常
// - 如果对象是 null 或空字符串 → 返回 defaultV（这是明确的预期行为）

Integer result = ValUtil.toInt("123");     // ✅ 返回 123
Integer result = ValUtil.toInt(null, 0);   // ✅ 返回 0（null 是预期的）
Integer result = ValUtil.toInt("", 0);     // ✅ 返回 0（空字符串是预期的）
Integer result = ValUtil.toInt("abc");     // ❌ 抛出异常（"abc" 不能转换为 int）
```

**3. 避免错误传播**

```java
// ❌ 静默失败导致错误传播
String userInput = "abc";  // 用户输入错误
Integer quantity = ValUtil.toInt(userInput, 0);  // 返回 0
int total = quantity * price;  // 计算错误的总价
order.setTotal(total);  // 保存错误数据
// 问题：错误数据进入系统，可能影响报表、统计、业务决策

// ✅ 快速失败阻止错误传播
String userInput = "abc";
try {
    Integer quantity = ValUtil.toInt(userInput);  // 立即抛出异常
    int total = quantity * price;
    order.setTotal(total);
} catch (NumberFormatException e) {
    // 在入口处处理错误，返回友好提示
    return "请输入有效的数量";
}
```

**4. 更容易调试**

```java
// ❌ 静默失败 - 难以定位问题
Integer value = ValUtil.toInt(data.get("amount"), 0);  // 返回 0
// ... 100 行代码后
if (value == 0) {
    // 这里出错了，但不知道是：
    // 1. amount 本来就是 0？
    // 2. amount 是 null？
    // 3. amount 是无效字符串？
    // 4. data 里没有 amount 字段？
}

// ✅ 快速失败 - 精确定位问题
Integer value = ValUtil.toInt(data.get("amount"));  
// 如果出错，异常堆栈直接指向这一行
// 开发者立即知道：data.get("amount") 返回了无效值
```

## 💻 ValUtil 的有界宽容实现

### 实现规则

1. **缺失（null/空）** → 返回 null 或默认值（缺失容忍）
2. **类型不同但能转换** → 自动转换（类型容忍）
3. **内容无法转换** → 抛出异常（内容不容忍）

### 代码示例

```java
// 1. 缺失容忍：null 和空字符串
ValUtil.toInt(null);           // → null（缺失容忍）
ValUtil.toInt(null, 0);        // → 0（缺失容忍，使用默认值）
ValUtil.toInt("");             // → null（缺失容忍）
ValUtil.toInt("", 0);          // → 0（缺失容忍，使用默认值）

// 2. 类型容忍：类型不同但能转换
ValUtil.toInt("123");          // → 123（String → Integer，类型容忍）
ValUtil.toInt(123L);           // → 123（Long → Integer，类型容忍）
ValUtil.toInt(123.45);         // → 123（Double → Integer，类型容忍）
ValUtil.toBoolean("true");     // → true（String → Boolean，类型容忍）
ValUtil.toBoolean(1);          // → true（Integer → Boolean，类型容忍）

// 3. 内容不容忍：内容无法转换
ValUtil.toInt("abc");          // ❌ NumberFormatException（内容不容忍）
ValUtil.toInt("xyz");          // ❌ NumberFormatException（内容不容忍）
ValUtil.toDouble("hello");     // ❌ NumberFormatException（内容不容忍）
```

## 何时使用默认值？

默认值应该用于**明确的预期场景**，而不是用来掩盖错误：

```java
// ✅ 正确使用默认值
Integer pageSize = ValUtil.toInt(request.getParameter("pageSize"), 10);
// 如果用户没有传 pageSize，使用默认值 10 是合理的

// ❌ 错误使用默认值
Integer userId = ValUtil.toInt(request.getParameter("userId"), 0);
// 如果 userId 是必需的，应该让它抛出异常，而不是使用默认值 0
// 正确做法：
Integer userId = ValUtil.toInt(request.getParameter("userId"));
if (userId == null) {
    throw new IllegalArgumentException("userId is required");
}
```

## 📝 总结

有界宽容原则的核心思想：

### 宽容的边界
- ✅ **缺失容忍**：路径不存在、值为 null、空字符串 → 返回 null 或默认值
- ✅ **类型容忍**：类型不同但能转换（"123" → 123）→ 自动转换
- ❌ **内容不容忍**：内容无法转换（"abc" → 123）→ 立即抛异常

### 设计优势
1. **宽容**：对预期内的情况（缺失、类型差异）保持宽容，减少样板代码
2. **有界**：对预期外的错误（内容错误）严格失败，及早发现问题
3. **明确**：边界清晰，行为可预测
4. **实用**：符合实际开发场景（API 数据不完整、前端类型混乱）

这种设计在宽容与严格之间找到了最佳平衡点，让代码既健壮又易用。
